## 생성형 AI

**Token**  
텍스트 처리의 기본 단위
- 정의
    - 텍스트를 구성하는 조각을 의미
    - 이 조각은 보통 단어를 나타내는데, 단어의 일부, 공백이나 구두점까지 포함될 수 있음
    - Tokenizer를 이용해 subword로 분해함

### LLM의 작동 방식

- 순차적인 텍스트를 입력으로 받아, 훈련된 데이를 기반으로 다음 토큰이 무엇이야 하는지 예측
- Given sequence로 next token을 auto-regressive 방식으로 예측

#### Prompt Engineering 이란?

- LLM이 정확한 결과를 생성하도록 안내하는 고품질 프롬프트를 설계하는 과정
- 이 과정에는 최적의 프롬프트를 찾기 위한 수정, 프롬프트 길이 최적화, 프롬프트의 글쓰기 스타일을 지정하는 작업이 포함됨
- 자연어 처리와 LLM의 맥락에서 프롬프트는 응답이나 예측을 생성하기 위해 모델에게 제공되는 입력임
- 사전학습이 완성된 LLM의 성능을 추가로 끌어올리는 방법
- 경량 모델(sLLM)에서는 잘 작동하지 않을 수 있음
- 동일한 Prompt이지만 모델마다 성능이 다르게 나타날 수 있음
- Prompt Engineering의 성능 상승은 선형적으로 증가하지 않음
- 비용 부담 없이 빠르게 해볼 수 있는 시도임

#### LLM 출력의 다양성 조정
1. Output length

다음에 생성할 토큰의 결정에 관여하는 설정들. 출력 길이에 맞는 Prompt 설계가 필요하다.
- 응답에서 생성할 토큰의 수
- 더 많은 토큰을 생성할수록 LLM에서는 더 많은 계산을 요구하고 이는 응답시간의 지연과 더 많은 비용을 발생시킴
- 출력 길이를 줄인다고 해서 LLM이 생성하는 출력의 스타일이나 텍스트가 더 간결해지는 것은 아님

2. Temperature

- 토큰 선택의 무작위성 정도를 제어
- 가장 낮은 값의 경우 가장 높은 확률의 토큰이 항상 선택되게 됨

3. Top-K

- 예측된 토큰 중 가장 가능성이 높은 상위 K 개의 토큰을 선택
- 값이 높을수록 모델의 출력은 더욱 창의적
- 값이 낮을수록 모델의 출력은 더욱 사실적

4. Top-p

- 누적 확률이 특정 값(p)를 초과하지 않는 상위 토큰을 선택
- p 값이 0일 경우, 가장 높은 확률의 토큰을 선택
- p 값이 1일 경우, LLM의 어휘집에 있는 모든 토큰이 출력 후보

5. Min-p

- 등장 확률이 너무 낮은 토큰은 제거
- p 값이 0.01이면 1% 미만의 등장 확률을 가진 토큰 제거

**reasoning.effort**  
최종 답변을 생성하기 전에 수행하는 내부 추론의 수준을 조절하는 옵션

|값|의미|적합한 작업|
|-|-|-|
|none|별도의 추론을 거의 수행하지 않음|분류, 단순 조회, 음성 응답, 빠른 챗봇|
|minimal|최소한의 추론|간단한 변환, 요약, 정형 작업|
|low|가벼운 추론|문서 작성, 데이터 분석, 간단한 코딩, 고객 응대|
|medium|균형 잡힌 추론|연구, 기획, 복합 분석, 에이전트 작업|
|high|깊은 추론|복잡한 디버깅, 전략 분석, 어려운 코딩|
|xhigh|매우 깊고 장시간의 추론|심층 연구, 보안, 코드 리뷰, 고난도 에이전트 작업|


<ins>왜 등장했는가?</ins>  

- 현대 LLM의 추론 능력이 크게 향상되면서 생긴 새로운 제어 장치(control knob)
- 문제 난이도마다 필요한 추론 냥이 다름
- 모든 문제를 최고 수준으로 추론하는 것은 비효율적
- 간단한 질문은 적은 추론, 복잡한 질문은 깊은 추론


**Verbosity**

- 최종 답변을 얼마나 간결하게 또는 상세하게 생성할지 조절하는 옵션
- 출력의 상세도를 제어하는 파라미터
- 핵심 내용은 같지만 설명의 범위와 세부 수준이 달라짐

|설정|출력 특징|적합한 용도|
|-|-|-|
|low|핵심만 짧게 답변|챗봇, 분류 결과, 임원 보고, 빠른 응답|
|medium|적당한 설명과 근거 포함| 일반 업무, 교육, 분석 보고서|
|high|배경, 과정, 예시까지 자세히 설명|강의 자료, 기술 설명, 심층 분석|


### Prompt의 종류

1. User Prompt

사용자가 모델에게 전달하는 입력
- 목적: 원하는 질문, 요청 또는 명령을 전달하여 모델이 적절한 출력을 생성하도록 유도
- 구체적이고 명확하게 작성하여 모델이 원하는 방향으로 응답하도록 유도

2. System Prompt

시스템이 모델에게 설정하는 초기 지침
- 목적: 모델의 동작 방식, 응답 스타일, 또는 역할 제어
- 모델의 역할과 맥락을 설정하여 일관성과 신뢰성 강화


### Prompt Engineering 실습
- 좋은 프롬프트는 질문을 자세히 쓰는 것이 아니라 구조적으로 작성하는 것
- RICE prompt는 AI에게 더 정확한 결과를 얻기 위해 질문을 4가지 요소로 구조화하는 방법
- 누가(Role), 무엇을(Instruction), 어떤 상황에서(Context), 어떤 방식으로(Examples)

#### Role(역할)
생성형 AI의 페르소나(역할)를 정의

> R(Role)은 모델에게 '누구처럼 생각할지'를 정해주는 것이고, Role을 부여하면 LLM은 해당 역할에 맞는 지식, 표현 방식, 판단 기준을 우선적으로 활용하여 응답을 생성

*하지만 최근 연구에서는 LLM에게 페르소나를 부여할 경우 페르소나 해석에 집중해 답변 퀄리티가 낮아진다고 하는 연구 또한 존재함*

#### Instruction(지시)
- 생성형 AI에게 무엇을 하라고 시키는지를 명확히 적는 부분
- 수행할 구체적 작업을 명시, 요청하는 질문이 불분명하면, AI도 모호한 답변을 할 가능성이 높음

> Instruction은 모델에게 "무엇을 생각하라"가 아니라 "어떤 절차로 판단하라"를 지정하는 것

#### Context(배경지식/정보제공)
- 질문과 관련하야 참고할 만한 지식과 정보를 함께 제공
- Hallucination을 줄이고 응답의 정확성과 관련성 높임
- 사고가 허용되는 경계 조건(boundary condition)을 설명

**LLM에게 제공할 Context를 모르겠다면...**  
<ins>Socratic Prompting</ins>  
- 소크라테스식 문답법을 프롬프트 설계에 적용하여 질문을 통해 비판적 사고와 추론을 유도하는 방식
- AI를 인터뷰어로 활용하는 기법
- 마지막에 "지금까지 내 답변을 바탕으로 제안서를 써줘"라는 프롬프트로 AI가 전체 대화를 컨텍스트로 활용하게 함

#### Examples(예시)
원하는 응답 형식이나 내용을 구체적인 예시를 제공하여 원하는 방향으로 응답 유도

> "Example은 출력 예시가 아니라 평가 기준의 표본. Example을 주는 순간 모델에게 채점 표를 건네는 것과 유사"

## Prompt Engineering Techniques

### 예시를 제공

#### General Prompting(zero-shot)
- 가장 간단한 유형의 프롬프트로 과제 설명 위주의 텍스트만 제공(질문, 지시 사항 등)
- zero shot은 예시 없음을 의미
- 간단하고 빠르게 적용 가능하고, 모델의 기본 추론 능력에 의존

#### One-shot
- 프롬프트를 작성할 때 예시를 제공하는 것
- 특정 출력 구조나 패턴으로 유도하려는 경우 유용하다
- 단 하나의 예시를 제공
- 모델이 작업을 가장 잘 완료하기 위해 모방할 수 있는 예시를 제공한다는 개념

#### Few-shot
- 모델에게 여러 가지 예시를 제공
- One-shot prompt와 유사하지만, 원하는 패턴의 여러 가지 예를 사용하면 모델이 패턴을 따른 확률이 높아짐
- 필요한 예제의 수는 작업의 복잡성, 예제의 품질 등 몇 가지 요인에 따라 달라지는데, 일반적으로 최소 3~5개의 예제를 사용
- 복잡한 작업의 경우 더 많은 예제를 사용해야 할 수도 있고, 모델의 입력 길이 제약으로 인해 더 적은 예제를 사용해야 할 수도 있음

### 중간 추론 과정 유도
#### Chain of Thought(CoT)
- 중간 추론 단계를 생성하여 LLM의 추론 능력을 향상시키는 기법으로 더욱 정확한 답변을 생성하는 데 도움이 됨
- 더 나은 결과를 얻기 위해 few-shot prompting과 함께 사용할 수 있음
- 문제를 더 작은 단계로 나누어 처리함으로써 복잡한 문제를 해결

<ins>장점</ins>  

- 매우 효과적이면서도 적은 노력으로 응답의 정확도 향상에 기여
- CoT 프롬프트를 사용하면 LLM의 추론 단계를 확인할 수 있어 해석 가능성도 높아지고, 잘못된 답변을 식별하기 용이함
- CoT는 여러 LLM 간의 전환 시에도 Robustness가 확보됨. CoT를 적용하지 않은 프롬프트애 비해 성능 변화 폭이 적음

#### Step-back Prompting
1. 과업과 관련된 일반적인 질문을 먼저 함  
    관련 배경지식과 추론 과정을 활성화
2. 필요한 질의를 후속 프롬프트에 입력
    수행 능력 향상 유도  

- 더 광범위하고 근본적인 원칙을 고려함으로써, 더욱 정확하고 통찰력 있는 답변을 유도
- 문제를 너무 빨리 구체화하거나 프레임 자체가 잘못된 상태에서 최적화하는 경우가 흔한데, Step-back Prompting은 질문의 전체, 암묵적 가정, 문제의 설정의 타당성을 흔들어보게 만드는 효과가 있음

### Self-Consistency Prompting
- CoT의 발전된 형태로, 복잡한 문제 해결에서 더 신뢰도 높은 답변을 얻기 위한 접근
- 하나의 질문에 대해 여러 번의 Chain-of-Thought 추론을 수행한 뒤, 그중 가장 많이 등장하는 결과를 최종 정답으로 선택하는 방식
- 여러 번의 질의 → reasoning path에 따른 서로 다른 답변 생성 → 가장 많이 등장하는 답변 선택(voting)

<ins>장점</ins>

- CoT는 탐욕적 디코딩 전략을 사용하기 때문에 효과가 제한적이고 실수하면 되돌릴 수 없음
- 샘플링과 다수결 투표를 결합하여 다양한 추론 경로를 생성하고 가장 일관된 답을 선택함으로써 LLM에서 생성되는 응답의 정확성과 일관성을 향상시킴

### 비판가 소환, Devil's Advocate Prompting
- 실행 전에 잠재적 실패를 먼저 예측하고 대안을 준비하는 방식으로 LLM의 성능을 향상
- 심리학의 "Pre-mortem"에서 영감
- "뭐가 잘못될 수 있어?"라고 물으면 보통 조심스럽고 두루뭉술한 답을 하지만, "이미 실패했어, 왜인지 말해봐"라고 하면 훨씬 구체적이고 솔직한 이유들이 도출됨
- 이미 실패했다고 가정하는 순간, AI가 훨씬 날카롭고 구체적인 반론을 생성

### 멀티 페르소나, Role-Based Multi-Persona Prompting
- 하나의 AI에게 서로 다른 입장의 역할을 동시에 부여해 충돌과 반박을 통해 의사결정의 사각지대를 찾는 기법
- AI 레드팀 토론: 내 아이ㄷ어를 AI가 먼저 공격하게 만드는 것
- 주장/기술 검증 시 AI를 활용
- 과도한 동의가 아닌 적절한 수준의 불일치가 최고의 성과를 낸다고 알려져 있음
- Multi Persona 기반 debate는 수학적 추론, 팩트체킹, 의사결정, 코드 분석 등 다양한 영역에서 성능 향상이 검증

### Meta Prompting
- AI에게 직접 답을 시키는 것이 아니라, 더 좋은 사고방식, 문제 해결 방식, 프롬프트를 스스로 구성하게 하는 기법
- 최종 답변을 얻는 게 아니라 "좋은 질문" 자체를 생성
- (1) 프롬프트 생성, (2) 사고 구조 설계
- 핵심 패턴은 Role 부여, 사고 구조 지정, 평가 기준 정의, 출력 구조 정의

> Meta Prompting은 AI에게 답을 묻는 것이 아니라, 연구자(사용자)의 사고 프레임워크를 생성하게 하는 기술

## 마크다운(Markdown) 언어란?
- 마크업 언어(Markup Language)는 문서의 구조나 의미를 태그나 기호로 표현하는 언어.
- Markdown은 마크업 언어의 일종으로 가볍고 읽기 쉬운 경량 마크업 언어
- 태크 대신 간단한 기호를 사용함

<ins>마크 다운을 사용하는 이유?</ins>  

- 좋은 프롬프트는 구조화된 프롬프트를 의미하며 마크다운 언어는 구조화를 가져가기 적절한 표현법: 의미적인 구분이 명확'
- LLM이 학습 과정에서 Markdown 형식의 문서를 매우 많이 접했기 때문에, 구조화된 Markdown 프롬프트가 더 안정적으로 해석되는 경향이 있음
- 공개된 고품질의 프롬프트 대부분은 Markdown으로 작성되어 있음: 좋은 프롬프트의 공통 패턴을 가져다 쓰기 용이
